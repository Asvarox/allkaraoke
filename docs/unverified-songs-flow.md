# Unverified Songs Flow (Cloudflare KV + PostHog + CI)

This document explains how the Unverified Songs pipeline works end-to-end, and how to set up KV namespaces and secrets for local, preview, and production environments.

## 1) What the flow does

Shared songs are submitted as PostHog events. CI validates and normalizes those shared songs, then upserts the playable candidates into Cloudflare KV as unverified songs.

At runtime, the app:

- Searches unverified songs via `GET /unverified-songs`
- Loads a selected unverified song via `GET /unverified-song?id=<sharedSongId>`
- Shows unverified songs as a fallback group in Song Selection when regular search results are low
- Provides a hidden browser admin panel at `/admin` for listing, deleting, reindexing, and editing unverified songs

## 2) Main components

- Storage contract and KV index: `functions/unverified-songs-store.ts`
- Public search endpoint: `functions/unverified-songs.ts`
- Public fetch-by-id endpoint: `functions/unverified-song.ts`
- Admin mutating endpoint (token-protected): `functions/unverified-songs-admin.ts`
- Browser admin endpoints (password-protected): `functions/admin/unverified-songs.ts`, `functions/admin/unverified-song.ts`
- Browser admin page: `src/routes/admin/admin.tsx`
- CI sync script (PostHog -> KV): `scripts/cicd/promote-shared-songs-to-unverified.ts`
- Admin API client for scripts: `scripts/cicd/unverified-songs-admin-client.ts`
- Fixture upsert script for local testing: `scripts/cicd/upsert-unverified-song-fixture.ts`

## 3) Data model stored in KV

Each unverified song record contains:

- `sharedSongId`
- `songId`
- `songTxt`
- `artist`
- `title`
- `language`
- `videoId`
- `validatedAt`
- `firstSeenAt`
- `lastSeenAt`
- `sourceUserId`
- `sourceEventAt`

Notes:

- Upsert semantics are full replacement for the record.
- Record keys are intentionally still stored under the legacy `shared-song:<sharedSongId>` prefix.
- The KV index is intentionally still stored at the legacy `shared-songs-index` key.
- The unverified-song index stores `sharedSongId` separately from `songId`, plus `firstSeenAt` for admin listing and sorting.
- Admin edits keep `sharedSongId` stable while updating KV content, even if corrected metadata produces a new `songId`.
- Listing/search use the KV-backed index; the index can be regenerated by scanning keys with the legacy `shared-song:` prefix.

## 4) End-to-end lifecycle

### 4.1 Intake

Users trigger PostHog events:

- `share-song` (contains song payload)
- `unshare-song` (contains song id)

### 4.2 CI sync and validation

Workflow: `.github/workflows/promote-shared-songs-to-unverified.yml`

The sync job:

1. Queries PostHog events in a time window
   Default scheduled/manual no-input run: last 14 hours
   Manual override: `hours_from` and `hours_to`
2. Validates parsed song structure and required metadata
3. Normalizes ESC suffix in titles when applicable
4. Validates lyric timing against YouTube duration (with fallback cap)
5. Calls admin endpoint:
   - `POST /unverified-songs-admin` to upsert
   - `DELETE /unverified-songs-admin?id=<id>` to remove
   - `PUT /unverified-songs-admin` to regenerate the index

The GitHub Actions schedule runs twice a day.

### 4.3 Runtime usage in app

- Search API: `src/modules/songs/unverified-songs/api.ts` -> `/unverified-songs`
- Song selection merge logic: `src/routes/sing-a-song/song-selection/hooks/use-song-list.ts`
- Unverified songs are presented in group: `Unverified songs`

### 4.4 Browser admin usage

The hidden admin panel is available at `/admin`. It is not linked from the main menu.

The browser admin flow:

1. Admin enters the browser admin password.
2. Frontend stores it in `sessionStorage` under `admin-panel-password`.
3. Frontend sends it to `/admin/*` endpoints as `x-admin-panel-password`.
4. Admin can list, filter, sort, delete, regenerate the index, and open an unverified song in the existing editor.
5. Admin edit saves update local song storage, share the corrected song locally, then update the stable KV record by `sharedSongId`.

Browser admin endpoints:

- `GET /admin/unverified-songs` lists index entries.
- `DELETE /admin/unverified-songs?id=<sharedSongId>` removes an unverified song.
- `PUT /admin/unverified-songs` regenerates the index.
- `PUT /admin/unverified-song?id=<sharedSongId>` updates an existing unverified song record.

## 5) KV namespace setup (prod + preview)

The project expects this binding in `wrangler.jsonc`:

- `binding = "SHARED_SONGS_KV"`
- `id = "<production namespace id>"`
- `preview_id = "<preview namespace id>"`

### 5.1 Create production namespace

Run:

```bash
pnpm wrangler kv namespace create SHARED_SONGS_KV
```

Copy the returned namespace `id` into `wrangler.jsonc` under `kv_namespaces` as `id`.

### 5.2 Create preview namespace

Run:

```bash
pnpm wrangler kv namespace create SHARED_SONGS_KV --preview
```

Copy the returned namespace `id` into `wrangler.jsonc` as `preview_id`.

### 5.3 Verify bindings

Run:

```bash
pnpm wrangler kv namespace list
```

Confirm both production and preview namespaces exist and match `wrangler.jsonc`.

## 6) Secrets management

### 6.1 Cloudflare runtime secret (CI admin endpoint auth)

Secret name:

- `UNVERIFIED_SONGS_ADMIN_TOKEN`

Set it for the deployed CI-oriented admin endpoint (`/unverified-songs-admin`).

If deployment is managed as a Worker:

```bash
pnpm wrangler secret put UNVERIFIED_SONGS_ADMIN_TOKEN
```

If deployment is managed as a Pages project:

```bash
pnpm wrangler pages secret put UNVERIFIED_SONGS_ADMIN_TOKEN --project-name allkaraoke-party
```

Guidelines:

- Do not commit this value to repo files.
- Rotate periodically and immediately if leaked.
- Keep local and CI values aligned with the environment being targeted (`gh secret set UNVERIFIED_SONGS_ADMIN_TOKEN`)

### 6.2 Cloudflare runtime secret (browser admin auth)

Secret name:

- `ADMIN_PANEL_PASSWORD`

Set it for the deployed browser admin endpoints under `/admin/*`.

If deployment is managed as a Worker:

```bash
pnpm wrangler secret put ADMIN_PANEL_PASSWORD
```

If deployment is managed as a Pages project:

```bash
pnpm wrangler pages secret put ADMIN_PANEL_PASSWORD --project-name allkaraoke-party
```

Guidelines:

- Do not reuse `UNVERIFIED_SONGS_ADMIN_TOKEN`; keep browser-admin and CI credentials separate.
- Do not commit this value to repo files.
- Send it only as the `x-admin-panel-password` request header.

### 6.3 Local development secret

Use `.dev.vars` (gitignored), for example:

```env
UNVERIFIED_SONGS_ADMIN_TOKEN=your-local-token
ADMIN_PANEL_PASSWORD=your-local-browser-admin-password
```

For script-driven local fixture upserts, defaults are in `.env`:

- `UNVERIFIED_SONGS_ADMIN_URL=http://127.0.0.1:3010`
- `UNVERIFIED_SONGS_ADMIN_TOKEN=local-unverified-songs-admin-token`

You can override with `.env.local`. Legacy `SHARED_SONGS_ADMIN_URL` and `SHARED_SONGS_ADMIN_TOKEN` values are still accepted as fallbacks.

### 6.4 GitHub Actions secrets

Shared songs workflows rely on these repo secrets:

- `UNVERIFIED_SONGS_ADMIN_URL`
- `UNVERIFIED_SONGS_ADMIN_TOKEN`
- `POSTHOG_PAT_KEY`

Additional import workflow secrets you likely already have:

- `GH_PAT_REPOS_WORKFLOW`
- `DEFAULT_IMPORT_USER_IDS`

Where used:

- `.github/workflows/promote-shared-songs-to-unverified.yml`
- `.github/workflows/import-songs.yml`

## 7) Bootstrapping checklist for a new environment

1. Create KV namespace for production and preview.
2. Put resulting IDs into `wrangler.jsonc` (`id` and `preview_id`).
3. Set runtime secrets `UNVERIFIED_SONGS_ADMIN_TOKEN` and `ADMIN_PANEL_PASSWORD` for Cloudflare.
4. Set GitHub repository secrets (`UNVERIFIED_SONGS_ADMIN_URL`, `UNVERIFIED_SONGS_ADMIN_TOKEN`, `POSTHOG_PAT_KEY`).
5. Run local fixture upsert to validate:

```bash
pnpm unverified-song:upsert-fixture
```

Optional: override the admin base URL directly from CLI (useful when test target differs from local default):

```bash
pnpm unverified-song:upsert-fixture http://127.0.0.1:3000
pnpm unverified-song:upsert-fixture https://localhost:3010
```

6. Verify APIs manually:

```bash
curl -s "http://127.0.0.1:8788/unverified-songs?query=Cloudflare%20Shared%20Unique%20Song"
curl -s "http://127.0.0.1:8788/unverified-song?id=shared-cloudflare-e2e-song"
```

## 8) Operational notes

- `GET /unverified-songs` requires `query` and supports `limit` (capped at 25).
- `GET /unverified-song` requires `id`.
- `POST /unverified-songs-admin` validates payload shape and upserts.
- `DELETE /unverified-songs-admin?id=<id>` removes song and index entry.
- `PUT /unverified-songs-admin` regenerates the index for CI/local scripts.
- `/admin/*` endpoints require `x-admin-panel-password`.
- The `/admin` frontend keeps the browser admin password in `sessionStorage` only.
- If neither `UNVERIFIED_SONGS_KV` nor legacy `SHARED_SONGS_KV` binding is configured, endpoints return 500 with a configuration error.

## 9) Troubleshooting

- `401 Unauthorized` on CI admin endpoint:
  - Check `x-unverified-songs-admin-token` header value against runtime secret.
- `401 Unauthorized` on browser admin endpoint:
  - Check `x-admin-panel-password` header value against `ADMIN_PANEL_PASSWORD`.
- `Unverified songs storage is not configured`:
  - Verify `UNVERIFIED_SONGS_KV` or legacy `SHARED_SONGS_KV` binding and namespace IDs in `wrangler.jsonc`.
- CI sync not writing data:
  - Verify GitHub secrets and target admin URL.
  - Check workflow logs in `.github/workflows/promote-shared-songs-to-unverified.yml`.
- Local fixture script fails env checks:
  - Ensure `.env` or `.env.local` has `UNVERIFIED_SONGS_ADMIN_TOKEN`.
  - Ensure admin URL is available either from `.env`/`.env.local` (`UNVERIFIED_SONGS_ADMIN_URL`) or as the first CLI argument.
